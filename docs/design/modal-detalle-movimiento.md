# Plan: Modal de Detalle de Movimiento (Read-Only)

## Contexto

Actualmente, para verificar los datos de un movimiento guardado (especialmente SPLIT con participantes), el usuario debe abrir el formulario de edición completo (`/registrar-movimiento?tipo=GASTO&edit={id}`), lo cual es pesado. La propuesta: al hacer click en un movimiento en la pestaña Gastos, se abre un modal read-only mostrando todos los campos, con botones de Editar y Eliminar al final.

## Decisiones

- **Trigger:** Click en la fila del movimiento (excepto el botón ⋮, que sigue funcionando como siempre)
- **Menú ⋮:** Se mantiene como atajo rápido para Editar/Eliminar sin abrir el modal
- **Alcance:** Solo pestaña Gastos (Phase 1). Extensible a Préstamos e Ingresos después.
- **Backend:** No requiere cambios. El `ListByHousehold` ya devuelve todos los campos incluyendo participantes de SPLIT. El modal se alimenta de los datos ya en memoria.
- **Tipos visibles en Gastos:** HOUSEHOLD y SPLIT (los SPLIT con categoría "Préstamo" se filtran y aparecen en la pestaña Préstamos, así que el modal en Gastos nunca verá DEBT_PAYMENT ni SPLIT-préstamo).

## Cambios

### 1. `frontend/pages/home.js` — Nueva función `showMovementDetailModal(movement)`

Crear una función que reciba el objeto `movement` (ya disponible en el array de la pestaña Gastos) y renderice un modal read-only con animación fade-in/slide-up.

**Tipos de movimiento en la pestaña Gastos:**

Solo aparecen estos dos tipos (los DEBT_PAYMENT y SPLIT con categoría "Préstamo" están en la pestaña Préstamos):
- **HOUSEHOLD** — Gasto del hogar normal
- **SPLIT** — Gasto compartido (con participantes)

**Contenido del modal por tipo:**

| Campo | HOUSEHOLD | SPLIT |
|-------|-----------|-------|
| Descripción | ✓ | ✓ |
| Monto (formateado COP via `formatCurrency()`) | ✓ | ✓ |
| Fecha (formateada via `formatDate()`, ej: "28 Mar 2026") | ✓ | ✓ |
| Categoría (`category_group_icon` + `category_name`) | ✓ | ✓ |
| Quién pagó (`payer_name`) | ✓ | ✓ |
| Método de pago (`payment_method_name`) | ✓ (si no null) | ✓ (si no null) |
| Participantes (nombre, porcentaje, monto) | — | ✓ |
| Badge "Compartido" (`movement.is_split === true`) | — | ✓ |
| Badge ⚡ Auto (`movement.generated_from_template_id && !movement.payer_user_id && !movement.payer_contact_id`) | si aplica | si aplica |
| Badge 📥 Bolsillo (`source_pocket_name`) | si aplica | — |

**Detalle de formateo de datos:**

- **Monto:** Usar `formatCurrency(movement.amount)` (ya existe en home.js, retorna formato COP: `$350.000`)
- **Fecha:** Usar `formatDate(movement.movement_date)` (ya existe en home.js, retorna: `28 Mar 2026`)
- **Categoría:** `${movement.category_group_icon} ${movement.category_name}` (ej: `🎬 Entretenimiento`)
- **Porcentaje participantes:** El campo `participant.percentage` viene como decimal (0.0-1.0), convertir a porcentaje: `${Math.round(participant.percentage * 100)}%`
- **Monto participantes:** Si `participant.amount` existe, usar `formatCurrency(participant.amount)`. Si no, calcular `formatCurrency(movement.amount * participant.percentage)`.
- **Auto-generado:** La condición exacta (consistente con la existente en el render de filas) es: `movement.generated_from_template_id && !movement.payer_user_id && !movement.payer_contact_id`

### 2. `frontend/pages/home.js` — Click handler en filas de movimientos

En las dos vistas (agrupada y cronológica), agregar click handler en las filas de movimientos.

**Vista agrupada:** `.movement-detail-entry` → click handler
**Vista cronológica:** `.chronological-movement-card` → click handler

**IMPORTANTE:** Estos handlers deben agregarse dentro del flujo de `setupCategoryListeners()` (o donde se enganchan los handlers del ⋮), para que se re-enganchen automáticamente cuando las filas se re-rendericen al cambiar de mes o aplicar filtros.

```javascript
// Dentro de setupCategoryListeners() o equivalente — se re-ejecuta al re-renderizar
document.querySelectorAll('.movement-detail-entry, .chronological-movement-card').forEach(row => {
  row.addEventListener('click', (e) => {
    // No abrir modal si el click fue en el botón ⋮ o en el menú
    if (e.target.closest('.three-dots-btn') || e.target.closest('.three-dots-menu')) return;

    const movementId = row.dataset.movementId;
    const movement = findMovementById(movementId);
    if (movement) showMovementDetailModal(movement);
  });
});
```

**Cambio necesario:** Agregar `data-movement-id="{id}"` a los contenedores `.movement-detail-entry` y `.chronological-movement-card` (actualmente solo el botón ⋮ tiene el `data-movement-id`).

**Función `findMovementById(id)`:** Buscar en `originalMovementsData.movements` (los datos sin filtrar) para que funcione incluso cuando hay filtros de categoría activos. Fallback a `movementsData.movements` si `originalMovementsData` no existe.

```javascript
function findMovementById(id) {
  const dataSource = originalMovementsData || movementsData;
  return dataSource?.movements?.find(m => m.id === id);
}
```

> **Nota:** Este mismo patrón (`originalMovementsData || movementsData`) ya se usa en `handleEditMovement()` y `handleDeleteMovement()`.

### 3. `frontend/pages/home.js` — Cerrar modal

```javascript
function closeMovementDetailModal() {
  const modal = document.getElementById('movement-detail-modal');
  if (modal) {
    // Animación de salida
    modal.classList.remove('modal-animate-in');
    modal.classList.add('modal-animate-out');
    const inner = modal.querySelector('.modal');
    if (inner) {
      inner.classList.remove('modal-slide-up');
      inner.classList.add('modal-slide-down');
    }
    // Esperar a que termine la animación antes de remover
    modal.addEventListener('animationend', () => modal.remove(), { once: true });
    // Fallback: remover después de 300ms si animationend no dispara
    setTimeout(() => { if (modal.parentNode) modal.remove(); }, 300);
  }
  // Remover listener de Escape
  document.removeEventListener('keydown', handleEscapeKey);
}

function handleEscapeKey(e) {
  if (e.key === 'Escape') closeMovementDetailModal();
}

// Dentro de showMovementDetailModal():
// Cerrar al hacer click fuera
overlay.addEventListener('click', (e) => {
  if (e.target === overlay) closeMovementDetailModal();
});
// Cerrar con Escape (registrar con referencia nombrada para poder removerlo)
document.addEventListener('keydown', handleEscapeKey);
```

## Edge Cases

| # | Caso | Decisión |
|---|------|----------|
| 1 | Movimiento con `source_pocket_id` (linked a bolsillo) | Mostrar badge "📥 {pocket_name}" en el modal. El botón Editar delega a `handleEditMovement()` que ya muestra el aviso "Editarlo desde Ahorros" si `source_pocket_id` existe. |
| 2 | Movimiento auto-generado (⚡ Auto) | Mostrar badge si `generated_from_template_id && !payer_user_id && !payer_contact_id`. Editar funciona normal. |
| 3 | SPLIT sin participantes cargados | No debería pasar — `ListByHousehold` ya carga participants para SPLIT. Si por algún edge case llega vacío, no mostrar la sección de participantes. |
| 4 | Campos nulos | Ocultar campos que no aplican (ej: `payment_method_name === null` → no mostrar esa fila). |
| 5 | Descripción muy larga | `.detail-value` tiene `word-wrap: break-word` y `max-width: 60%` para permitir wrapping. |
| 6 | Click en ⋮ dentro de la fila | El `e.stopPropagation()` existente en el handler del ⋮ ya previene que el click suba. El nuevo handler además verifica con `e.target.closest('.three-dots-btn')`. |
| 7 | Movimiento eliminado por otro usuario | `handleDeleteMovement()` ya maneja el error. Si falla (404), muestra error toast. |
| 8 | Re-renderizado de filas (cambio de mes, filtros) | Los click handlers se agregan en el mismo flujo de setup que los handlers del ⋮, así se re-enganchan automáticamente al re-renderizar. |
| 9 | Listener de Escape colgado | `closeMovementDetailModal()` remueve explícitamente el listener de keydown usando una referencia nombrada (`handleEscapeKey`). |
| 10 | Participante con `amount` null | Si `participant.amount` es null, calcular como `movement.amount * participant.percentage` y formatearlo con `formatCurrency()`. |

## Verificación

1. Click en un movimiento HOUSEHOLD → modal muestra todos los campos, sin sección de participantes
2. Click en un movimiento SPLIT → modal muestra participantes con nombre, porcentaje (ej: "50%") y monto
3. Click en ⋮ → NO abre el modal, sigue abriendo el menú como antes
4. Botón Editar en modal → navega al formulario de edición (delega a `handleEditMovement`)
5. Botón Eliminar en modal → muestra confirmación de eliminación (delega a `handleDeleteMovement`)
6. Click fuera del modal / Escape / ✕ → cierra el modal con animación fade-out
7. Movimiento auto-generado → muestra badge ⚡ Auto
8. Movimiento de bolsillo → muestra badge 📥 {pocket_name}, Editar muestra aviso
9. Vista agrupada y cronológica → ambas abren el modal
10. Cambiar de mes → re-renderiza filas → click en fila sigue abriendo modal
11. Aplicar filtro de categorías → click en fila visible sigue abriendo modal
12. Modal aparece con animación fade-in + slide-up, cierra con fade-out + slide-down
13. Descripción larga → texto hace wrap en el modal sin desbordarse
